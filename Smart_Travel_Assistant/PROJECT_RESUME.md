# 智旅 AI — 产品项目说明书（简历版）

> 项目定位：面向自助旅行者的 AI 智能旅行规划平台
> 技术栈：Python · ReAct Agent · LLM (OpenAI Compatible) · Flask · Folium
> 项目规模：~1200 行核心代码 · 6 个核心模块 · 2 种交互形态（CLI + Web）

---

## 一、产品概述

**智旅 AI** 是一款基于大语言模型的智能旅行规划系统，覆盖从「灵感激发 → 行程生成 → 预算规划 → 天气适配」的全流程。产品采用 **ReAct Agent 架构**实现自主推理与多工具协同调度，同时提供 **Web 可视化界面**与 **CLI 命令行**两种交互形态，满足不同场景下的使用需求。

### 目标用户

| 用户画像 | 核心痛点 | 使用场景 |
|---------|---------|---------|
| 自助游爱好者 | 做攻略耗时、信息分散 | 输入目的地 + 天数，快速获得完整行程 |
| 商务出行人员 | 时间紧、需要高效规划 | 一键生成商务城市短期行程 |
| 家庭出游用户 | 需兼顾老人/小孩需求 | 通过偏好标签筛选适合亲子的景点 |

## 二、产品功能矩阵

### 2.1 AI 行程生成引擎（核心功能）

| 维度 | 说明 |
|------|------|
| 产品价值 | 将用户从繁琐的攻略制作中解放出来，输入目的地 + 天数即可获得结构化行程 |
| 输入方式 | Web 端表单输入（目的地文本 + 天数步进器 + 偏好标签多选） |
| 核心流程 | 用户输入 → Tavily 搜索景点 → LLM 结构化输出 → 格式化渲染 |
| 输出内容 | 每日主题 · 景点列表 · 描述 · 类别标签 · 建议时长 · 实用贴士 |
| 差异化 | 支持 6 类偏好标签精准控制内容倾向，非泛泛推荐 |
| 技术实现 | 通过 System Prompt 约束 LLM 输出固定 JSON Schema，前端直接渲染 |

**用户故事**：
> 作为一名计划去北京旅游 3 天的用户，我希望选择「经典 + 美食」偏好，这样系统能为我规划一条涵盖故宫、天安门等经典景点，同时穿插南锣鼓巷等美食街的行程路线。

### 2.2 智能天气适配模块

| 维度 | 说明 |
|------|------|
| 产品价值 | 帮助用户根据实时天气调整行程，避免雨天去户外景点 |
| 数据源 | wttr.in 公共天气 API |
| 覆盖范围 | 50+ 城市时区自动映射，支持模糊匹配（如「北京市」到「北京」）|
| 容错机制 | 时区查询失败自动降级使用本地时间 |

### 2.3 旅行预算规划引擎

| 维度 | 说明 |
|------|------|
| 产品价值 | 让用户在出行前对花费有清晰预期，辅助决策目的地和天数 |

### 2.4 偏好标签系统

| 维度 | 说明 |

| 交互方式 | 点击切换选中态，可多选组合 |
| 推荐策略 | 标签作为 LLM Prompt 的指令约束，引导模型重点推荐相关类别景点 |

### 2.5 双端交互形态

## 三、系统架构

### 3.1 架构层次

```
用户层         Web 浏览器                   CLI 终端
                  │                           │
接入层        Flask Server              ReAct Agent Loop
              (REST + 模板)          (Thought-Action循环)
                  │                           │
业务层      行程生成    天气查询    预算计算    地图生成
            引擎        模块        引擎        模块
                  │       │           │          │
工具层       available_tools 动态注册与调度
                  │
基础层      OpenAI API · Tavily · wttr.in · Geopy
```

### 3.2 模块职责

| 模块 | 文件 | 职责 |
|------|------|------|
| Agent 主循环 | main.py | ReAct 框架实现，管理 Thought-Action-Observation 循环 |
| Web 服务 | app.py | Flask 路由、API 端点、请求响应处理 |
| 行程生成 | itinerary_generator.py | LLM 调度 + Tavily 搜索 + 结构化输出 |
| LLM 客户端 | llm.py | OpenAI Compatible API 封装 |
| 工具函数 | tools.py | 天气查询、景点搜索、预算计算具体实现 |
| 提示词模板 | prompt_template.py | Agent System Prompt 设计与维护 |
| 数据映射 | mapping_relation.py | 50+ 城市时区映射表 |
| 前端模板 | templates/index.html | 自包含 HTML/CSS/JS 单页应用 |

### 3.3 核心数据流（Web 行程生成）

```
用户输入（目的地 + 天数 + 偏好标签）
      │
      ▼
Tavily Search API ──── 实时搜索景点攻略信息
      │
      ▼
构造 LLM Prompt ──── System Prompt 包含 JSON Schema 约束
      │
      ▼
LLM（通义千问 / GPT 等兼容模型）
      │
      ▼
输出结构化 JSON ──── 解析 / 校验 / 容错降级
      │
      ├── Folium 生成交互式地图 HTML
      └── 格式化文本行程（卡片式每日展示）
                  │
                  ▼
           返回前端渲染
```

## 四、技术亮点与难点

### 4.1 ReAct Agent 框架实现

**难点**：LLM 输出的 Thought-Action 对可能存在换行、截断、重复等问题，需要精确解析。

**解决方案**：
- 使用前瞻断言正则表达式精准提取第一个 Thought-Action 对
- 超出部分自动截断，防止模型「幻觉」输出干扰后续步骤
- 工具参数通过正则动态提取，支持任意参数数量的工具调用

### 4.2 结构化行程输出的可靠性

**难点**：LLM 可能输出格式不规范的 JSON，导致前端解析失败。

**解决方案**：三级解析策略——
1. 标准 json.loads 直接解析
2. 提取 Markdown 代码块后解析
3. 提取花括号范围后解析

### 4.3 多源数据融合

**难点**：天气数据、景点搜索、预算数据来自不同 API 和数据源，需要融合为统一的输出。

**解决方案**：Agent 的 Thought-Action 循环按需依次调用各工具，结果累积在 Prompt History 中，LLM 最终综合所有信息给出完整回答，保证信息的一致性。

### 4.4 前端的零依赖设计

**亮点**：Web 前端采用单一 HTML 文件，不依赖任何构建工具或 npm 包。
- 所有 CSS 和 JavaScript 内联在模板中
- 使用原生 Fetch API 与后端通信
- 自适应桌面和移动端布局
